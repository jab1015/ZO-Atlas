# ATLAS PRODUCT VISION & UX BIBLE v1.0

## Purpose

This document defines the **user experience philosophy** for Atlas. It
is the first document designers and developers should read before
implementing features.

**Technical specifications describe how Atlas works.** **This document
describes how Atlas should feel.**

------------------------------------------------------------------------

# Product Vision

Atlas is not project management software.

Atlas is an intelligent guide that helps inventors move from **idea to
market** with confidence.

The user should never feel like they are operating software. They should
feel like they are working with an experienced invention coach.

------------------------------------------------------------------------

# Design Principles

## 1. Simplicity First

Every screen should answer one question:

**"What's the next best thing I should do?"**

Avoid clutter and unnecessary choices.

------------------------------------------------------------------------

## 2. Hide Complexity

Users should not see:

-   Engine
-   AI Agents
-   Readiness algorithms
-   API concepts
-   Security rules
-   Billing mechanics

Those are implementation details.

------------------------------------------------------------------------

## 3. Speak the Inventor's Language

Prefer:

-   Invention
-   Prototype
-   Next Step
-   Journey
-   Product
-   Market

Avoid technical project-management terms where possible.

------------------------------------------------------------------------

## 4. Guide, Don't Overwhelm

Atlas should present one recommended action at a time.

Large checklists and dashboards should be secondary.

------------------------------------------------------------------------

## 5. Celebrate Progress

Each completed stage should feel like a milestone.

Examples:

-   Validation Complete
-   Prototype Ready
-   Manufacturing Planned

The experience should encourage momentum.

------------------------------------------------------------------------

# Home Screen

After sign in, Atlas welcomes the user.

Example:

Welcome back.

What would you like to do today?

-   Continue my invention
-   Start a new invention
-   Learn something
-   Ask Atlas

------------------------------------------------------------------------

# Creating a New Invention

Instead of asking for a project name first, Atlas begins a conversation.

Suggested flow:

1.  What problem are you trying to solve?
2.  Who has this problem?
3.  Tell me about your idea.
4.  Do you have sketches or photos?
5.  Atlas creates the invention workspace.

------------------------------------------------------------------------

# Dashboard

The dashboard centers on inventions.

Each invention shows:

-   Current stage
-   Atlas recommendation
-   Continue button

Administrative features belong in Settings.

------------------------------------------------------------------------

# Navigation

Primary navigation:

-   My Inventions
-   Learn
-   Messages
-   Profile

Secondary:

-   Subscription
-   Billing
-   Settings
-   Support

------------------------------------------------------------------------

# Subscription Philosophy

Users pay to advance their invention, not to create additional projects.

Explorer (Free): - One active invention - Idea - Validation - Basic
Research

Inventor Pro: - Complete Inventor Journey - Full AI guidance - Advanced
planning tools

Enterprise: - Teams - Shared workspaces - Organization features

------------------------------------------------------------------------

# Tone of Voice

Atlas is:

-   Encouraging
-   Clear
-   Practical
-   Professional

Atlas avoids fear-based messaging and unnecessary jargon.

------------------------------------------------------------------------

# Success Metric

If a first-time inventor can open Atlas and immediately understand the
next step without training, the experience is successful.

------------------------------------------------------------------------

# Guiding Principle

**Every design decision should make inventing feel simpler, clearer, and
more achievable.**
